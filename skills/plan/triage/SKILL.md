# Triage

Move issues on your project tracker through a small state machine of triage roles.

Every comment or issue posted during triage **must** start with:

> *This was generated by AI during triage.*

## Reference

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — how to write durable agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — how the out-of-scope knowledge base works

## Roles

Two **category** roles:
- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** roles:
- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more information
- `ready-for-agent` — fully specified, ready for an AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Every triaged issue carries exactly one category role and one state role. If state roles conflict, flag it and ask before proceeding.

These are canonical role names — the actual label strings may differ per tracker. Run setup-agentforge if label mapping is not configured.

State transitions: unlabeled → `needs-triage` → `needs-info` / `ready-for-agent` / `ready-for-human` / `wontfix`. `needs-info` returns to `needs-triage` once the reporter replies. Flag unusual transitions and ask before proceeding.

## Invocation

The maintainer invokes triage in natural language:

- "Show me anything that needs attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## Show What Needs Attention

Query the issue tracker and present three buckets, oldest first:

1. **Unlabeled** — never triaged.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with reporter activity since last triage** — needs re-evaluation.

Show counts and one-line summary per issue. Let the maintainer pick.

## Triage a Specific Issue

1. **Gather context.** Read the full issue (body, comments, labels, reporter, dates). Parse prior triage notes. Explore the codebase using CONTEXT.md, respecting ADRs. Read `.out-of-scope/*.md` and surface any prior rejection that resembles this issue.

2. **Recommend.** State category and state recommendation with reasoning, plus a brief codebase summary. Wait for direction.

3. **Reproduce (bugs only).** Before grilling, attempt reproduction: read the reporter's steps, trace relevant code, run tests or commands. Report what happened — confirmed repro, failed repro, or insufficient detail (strong `needs-info` signal).

4. **Grill (if needed).** If the issue needs fleshing out, run grill-with-docs.

5. **Apply the outcome:**
   - `ready-for-agent` — post an agent brief comment (see AGENT-BRIEF.md).
   - `ready-for-human` — same structure, note why it can't be delegated.
   - `needs-info` — post triage notes (template below).
   - `wontfix` (bug) — polite explanation, then close.
   - `wontfix` (enhancement) — write to `.out-of-scope/`, link from comment, close.
   - `needs-triage` — apply the role. Optional comment if partial progress.

## Quick State Override

If the maintainer says "move #42 to ready-for-agent", apply directly. Confirm what you're about to do, then act. Skip grilling. If moving to `ready-for-agent` without grilling, ask whether they want an agent brief.

## Needs-Info Template

```
## Triage Notes

**What we've established so far:**
- point 1
- point 2

**What we still need from you (@reporter):**
- question 1
- question 2
```

Capture everything resolved under "established so far." Questions must be specific and actionable.

## Resuming a Previous Session

If prior triage notes exist, check whether the reporter answered outstanding questions. Present an updated picture before continuing. Don't re-ask resolved questions.
